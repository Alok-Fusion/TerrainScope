======================================================================
1. PROJECT OVERVIEW
======================================================================

Project name:
TerrainScope - Offroad Semantic Segmentation and Live Inference Dashboard

Project goal:
Build a complete off-road scene segmentation system that:
- trains a semantic segmentation model on the Falcon off-road dataset,
- predicts one of 10 terrain/object classes for each pixel,
- reports standard segmentation metrics such as IoU, Dice, and pixel accuracy,
- exports prediction artifacts and failure cases,
- and provides a professional frontend for visual review and live image inference.

End result:
The final project is not just a training script. It is an end-to-end submission kit with:
- a clean data contract,
- a config-driven training and evaluation pipeline,
- checkpointing and resume support,
- visual analysis outputs,
- a FastAPI backend,
- and a React frontend for run browsing and upload inference.


======================================================================
2. PROBLEM STATEMENT
======================================================================

The task is to semantically segment off-road scenes into meaningful classes that support
terrain understanding and traversability reasoning.

Why this matters:
- Off-road environments are visually complex.
- Different terrain types and obstacles often overlap in texture and color.
- Lighting, vegetation density, and clutter make pixel-wise classification difficult.
- Small or infrequent classes such as logs, rocks, or clutter are easy to miss.

The required output:
- A per-pixel class prediction over 10 semantic classes.
- Reliable evaluation using segmentation metrics.
- Visual evidence through color masks, comparison images, and failure analysis.

Core challenge:
Create a solution that is accurate enough to be useful, while also being practical to run
in a CPU-constrained environment during development.


======================================================================
3. SOLUTION SUMMARY
======================================================================

The implemented solution uses transfer learning with a frozen DINOv2 visual backbone and
a lightweight segmentation head trained on top of extracted patch tokens.

High-level solution choices:
- Use DINOv2 features instead of training a full segmentation network from scratch.
- Freeze the backbone to reduce training cost and make CPU training feasible.
- Train only a compact segmentation head.
- Enforce one shared 10-class label contract across training, evaluation, visualization,
  frontend, and backend inference.
- Save all artifacts needed for a submission:
  - checkpoints,
  - metrics,
  - plots,
  - predicted masks,
  - colorized masks,
  - comparison figures,
  - worst-case examples.
- Add a frontend and backend layer so the project can be demonstrated interactively.

Why this approach was chosen:
- DINOv2 provides strong general-purpose visual representations.
- Freezing the backbone drastically reduces compute cost.
- A custom lightweight head is easier to train in a CPU-safe setup.
- A strict dataset contract prevents silent label corruption.

6. LABEL CONTRACT AND MASK HANDLING
======================================================================

Canonical raw Falcon mask values:
- 100   -> Trees
- 200   -> Lush Bushes
- 300   -> Dry Grass
- 500   -> Dry Bushes
- 550   -> Ground Clutter
- 600   -> Flowers
- 700   -> Logs
- 800   -> Rocks
- 7100  -> Landscape
- 10000 -> Sky

Canonical class IDs:
- 0 -> Trees
- 1 -> Lush Bushes
- 2 -> Dry Grass
- 3 -> Dry Bushes
- 4 -> Ground Clutter
- 5 -> Flowers
- 6 -> Logs
- 7 -> Rocks
- 8 -> Landscape
- 9 -> Sky

Ignore index:
- 255

Why this is important:
- The masks are not standard 0..9 class IDs in raw form.
- They must be read without truncation.
- Reading them incorrectly can silently destroy labels.

Implementation details:
- Masks are read with cv2.IMREAD_UNCHANGED.
- Unknown raw values are mapped to 255.
- In strict mode, unknown values raise an error instead of silently turning into class 0.
- Mask resize uses nearest-neighbor interpolation to preserve class labels.

Verified dataset raw values:
- Train + validation raw values:
  [100, 200, 300, 500, 550, 600, 700, 800, 7100, 10000]
- Test raw values:
  [100, 200, 300, 500, 800, 7100, 10000]

Interpretation:
- The train and validation splits contain the full supported label contract.
- The test split contains a valid subset of the supported classes.


======================================================================
7. PREPROCESSING PIPELINE
======================================================================

RGB preprocessing:
- Read image as RGB
- Resize to model input resolution
- Convert to float in [0, 1]
- Normalize with:
  - mean = [0.485, 0.456, 0.406]
  - std  = [0.229, 0.224, 0.225]
- Convert to CHW tensor

Mask preprocessing:
- Read uint16/raw mask
- Convert raw values to class IDs
- Resize with nearest-neighbor interpolation
- Convert to long tensor

Quick CPU run input size:
- 140 x 252

Patch size:
- 14

Resulting token grid:
- 10 x 18

8. MODELS USED
======================================================================

Primary vision backbone:
- DINOv2 ViT-S/14
- Model name: dinov2_vits14

How the backbone is used:
- Loaded through torch.hub from facebookresearch/dinov2
- Set to eval mode
- All parameters frozen
- Used only as a feature extractor

Backbone output used:
- x_norm_patchtokens

Segmentation head:
- Custom ConvNeXt-style lightweight head
- Implemented in:
  Offroad_Segmentation_Scripts\offroad_segmentation\model.py

Segmentation head structure:
- 7x7 Conv2d stem to 128 channels
- GELU activation
- Depthwise 7x7 Conv2d block
- GELU activation
- Pointwise 1x1 Conv2d
- GELU activation
- Final 1x1 classifier to 10 classes

Why this model combination was used:
- DINOv2 gives strong pretrained features for visual understanding.
- Freezing the backbone reduces training cost.
- The lightweight head makes the system practical on CPU.
- The head can learn task-specific segmentation boundaries without updating the full backbone.

15. METRIC DEFINITIONS
======================================================================

IoU (Intersection over Union):
- Measures overlap between predicted and ground-truth regions.
- Formula:
  IoU = TP / (TP + FP + FN)
- This is one of the most important semantic segmentation metrics.

Dice coefficient:
- Measures overlap with stronger emphasis on matching regions.
- Formula:
  Dice = 2TP / (2TP + FP + FN)

Pixel accuracy:
- Fraction of correctly classified pixels over all valid pixels.

Why IoU and Dice matter here:
- Pixel accuracy can look strong even when rare classes are missed.
- IoU and Dice reveal whether boundaries and minority classes are actually learned.

19. CHALLENGES ENCOUNTERED
======================================================================

1. Training speed on CPU
Problem:
- Full-resolution or heavy models were too slow on CPU.

Fix:
- Freeze the DINOv2 backbone.
- Use a quick CPU config.
- Reduce image size to 140 x 252.
- Use batch size 1 and workers 0.
- Add resume support for longer runs.

Result:
- Training became feasible locally.

2. Mask-format issues
Problem:
- Falcon masks use nonstandard raw values and can be corrupted by incorrect reading.

Fix:
- Read masks unchanged.
- Convert through a strict shared mapping.

Result:
- Correct supervision signal for training and evaluation.

3. Dataset path mismatch
Problem:
- Initial scripts assumed simpler paths than the actual nested dataset layout.

Fix:
- Standardized exact dataset roots.

Result:
- Stable dataset discovery across train, validation, and test.

4. Inconsistent scripts
Problem:
- The previous code path was fragmented and not submission-ready.

Fix:
- Rebuilt the usable pipeline outside the legacy folder.

Result:
- Cleaner, reproducible project structure.

5. Demonstration quality
Problem:
- Static plots alone are not enough for presentation.

Fix:
- Added a frontend and API for live inference and guided review.

Result:
- The project is easier to present and inspect.

20. LIMITATIONS
======================================================================

Current limitations of the final model:
- Rare classes are weakly learned.
- Small clutter objects remain difficult.
- The backbone is frozen, so representation adaptation is limited.
- Training was intentionally CPU-safe rather than maximum-performance oriented.
- The quick config disables full raw-value scan during the final fast run to save time.
- The final run used capped batches per epoch for faster local iteration.

Observed weak classes:
- Dry Bushes
- Ground Clutter
- Logs
- Rocks
- Several of these had zero or near-zero IoU in evaluation

Meaning:
- The model is useful as a baseline and demo system, but not yet a high-performing
  production-grade off-road segmenter across all classes.
